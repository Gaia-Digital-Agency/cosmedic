## Rearrangement Guidelines

### D1 — Merge split titles and headings
Any field split into two parts (A/B, Part1/Part2, Accent) for a single semantic heading is merged into one field. CSS handles line breaks and italic accents at render time.

- `titleA + titleB` → `title`
- `headingA + headingB` → `heading`
- `headingPart1 + headingPart2` → `heading`
- `headingItalic + headingPart2` → `heading`
- `headingPart1 + headingAccent` → `heading`

**Exception:** `pullQuoteBefore / pullQuoteAccent / pullQuoteAfter` in HomeIntro — stay split because `pullQuoteAccent` has distinct semantic meaning (the one italic word), not just a line break.

### D2 — Rename fields whose name collides with a different meaning elsewhere
If the same field name is used for semantically different things across items, rename the one that differs to avoid confusion in the CMS admin.

- `title` in Surgeons collection (meaning honorific "dr.") → `designation`

### D3 — Hero standard field order
All Hero globals follow this fixed order:

`breadcrumbLabel, title, lede, heroImage, imageHue, imageLabel, chapter`

### D4 — Page reading order for content groups
Within any item, fields appear in the order an editor would encounter them reading the page top-to-bottom: eyebrow → heading → lede → body → CTAs → supporting fields.

### D5 — Group related fields together, never split groups
Fields that belong to the same section or functional area stay together. Sidebar labels stay together, stat labels stay together, breadcrumb labels stay together.

### D6 — Detail Template group order
For complex template globals, the group order follows the page reading sequence:

`breadcrumbs → hero labels → stat labels → sidebar labels → language/availability fallbacks → faculty → specialty → training → biography → orphan singles`

### D7 — Image fields second-last, SEO before orphan singles, orphan singles last
The tail of any field list follows this fixed sequence:

`[content fields] → [image fields] → seo → [orphan single items]`

- Image (upload) fields appear in the second-last position
- `seo` group always comes directly before orphan singles
- Orphan single items (e.g. `hue`, `sortOrder`) are last

### D8 — lede before eyebrow in section views
For section view globals (PSV, AMV and similar) where lede and eyebrow appear together, `lede` comes first because it describes the section purpose; `eyebrow` is a label detail.

`lede, eyebrow, heading, headingItalic`

### D9 — Remove duplicate hero fields from Main globals
Any page Main global that has a dedicated Hero global must use `pageFields({ hideHero: true })`. This removes `chapterTitle, tagline, lede, heroImage` from the Main panel so the same fields are not editable in two places.

Applies to: JourneyPage, ContactPage, ResultsPage (SurgeonsPage and PricingPage already done).

### D10 — UMTP no changes
Users, Media, Treatments, and Pricing bucket items are listed in the table as-is. No field merges, reorders, or config changes are applied to them.
